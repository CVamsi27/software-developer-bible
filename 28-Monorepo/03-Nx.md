# Nx

## Definition
Nx is a smart, fast, and extensible build system for monorepos. It provides tools for managing multiple projects, code generation, dependency analysis, and computation caching to improve developer experience and build performance.

## Why Do We Need It?

- **Code Generators**: Scaffold new projects, components, and features
- **Affected Commands**: Only run tasks for changed projects
- **Computation Cache**: Cache build results for faster rebuilds
- **Dependency Graph**: Visualize and analyze project dependencies
- **Extensible**: Plugin ecosystem for various frameworks

```typescript
// tools/generators/component/index.ts
import { Tree, formatFiles, names, joinPathFragments } from '@nx/devkit';
import * as path from 'path';

interface ComponentGeneratorOptions {
  project: string;
  name: string;
  export?: boolean;
}

export default async function (
  tree: Tree,
  options: ComponentGeneratorOptions
) {
  const projectRoot = tree.exists(options.project)
    ? // If it's a project name, find its root
      Object.entries(
        JSON.parse(tree.read('workspace.json', 'utf-8')!).projects
      ).find(([_, v]) => v === options.project)?.[0] || options.project
    : options.project;

  const componentName = names(options.name).className;
  const fileName = names(options.name).fileName;

  const templateOptions = {
    ...options,
    ...names(options.name),
    projectName: options.project,
    export: options.export ?? true,
  };

  // Generate component files
  await generateFiles(
    tree,
    path.join(__dirname, 'files'),
    joinPathFragments(projectRoot, 'src', 'lib', fileName),
    templateOptions
  );

  await formatFiles(tree);
}
```

```typescript
// tools/executors/analyze/index.ts
import { ExecutorContext } from '@nx/devkit';
import * as fs from 'fs';
import * as path from 'path';

interface AnalyzeExecutorOptions {
  budget: number;
}

export default async function* analyzeExecutor(
  options: AnalyzeExecutorOptions,
  context: ExecutorContext
) {
  const projectRoot = context.workspace.projects[context.projectName!].root;
  const outputPath = path.join(projectRoot, 'dist');

  // Calculate bundle size
  const bundleSize = calculateBundleSize(outputPath);

  console.log(`Bundle size: ${bundleSize}KB`);
  console.log(`Budget: ${options.budget}KB`);

  if (bundleSize > options.budget) {
    console.error(`Bundle exceeds budget by ${bundleSize - options.budget}KB`);
    return { success: false };
  }

  console.log('Bundle size within budget');
  return { success: true };
}

function calculateBundleSize(dir: string): number {
  let totalSize = 0;

  const files = fs.readdirSync(dir, { withFileTypes: true });
  for (const file of files) {
    const filePath = path.join(dir, file.name);
    if (file.isDirectory()) {
      totalSize += calculateBundleSize(filePath);
    } else {
      const stats = fs.statSync(filePath);
      totalSize += stats.size / 1024; // Convert to KB
    }
  }

  return Math.round(totalSize * 100) / 100;
}
```

### 5. Affected Commands

```bash
# Run build only for affected projects
nx affected --target=build

# Run tests for affected projects
nx affected --target=test

# Run lint for affected projects
nx affected --target=lint

# Compare against specific branch
nx affected --target=build --base=main

# Run multiple targets
nx affected --targets=build,test,lint

# Run with parallel execution
nx affected --target=build --parallel=3

```

## Common Mistakes

1. **Over-modularizing**: Creating too many small libraries

2. **Wrong dependency direction**: Apps depending on each other

3. **Ignoring tags**: Not using tags for boundary enforcement

4. **Not using generators**: Manual file creation instead of code generation

5. **Missing configuration**: Incomplete project.json files

## Best Practices

1. **Start simple**: Begin with fewer libraries, split when needed

2. **Use tags**: Enforce architectural boundaries

3. **Leverage generators**: Maintain consistency with code generation

4. **Monitor dependencies**: Use dependency graph regularly

5. **Cache everything**: Configure caching for all cacheable tasks

## Interview Questions

### Beginner (5)

1. **What is Nx?**

   - Answer: A smart, fast, and extensible build system for monorepos with code generation, affected analysis, and computation caching.

2. **What are Nx generators?**

   - Answer: Code generation tools that scaffold new projects, components, and features with consistent patterns.

3. **What is affected analysis?**

   - Answer: A feature that determines which projects are affected by changes and only runs tasks for those projects.

4. **What is computation caching?**

   - Answer: Caching of task results based on inputs, so tasks don't re-run if inputs haven't changed.

5. **What is the Nx dependency graph?**

   - Answer: A visualization of project dependencies, showing how projects relate to each other.

### Intermediate (5)

6. **How do you configure caching in Nx?**

   - Answer: Use `targetDefaults` in nx.json to specify which tasks are cacheable and their inputs/outputs.

7. **What are named inputs?**

   - Answer: Reusable input definitions in nx.json that can be referenced by targets.

8. **How do you enforce architectural boundaries?**

   - Answer: Use tags in project.json and configure linting rules to enforce dependency constraints.

9. **What is the difference between libs and apps?**

   - Answer: Apps are deployable applications; libs are shared libraries consumed by apps and other libs.

10. **How do you run tasks for specific projects?**

    - Answer: Use `nx <target> <project>` or `--projects` flag.

### Senior (10)
11. **Explain Nx's computation cache algorithm**

    - Answer: Content-based hashing of inputs (files, env, dependencies), compares with stored hash, returns cached outputs if match.

12. **How does Nx determine affected projects?**

    - Answer: Analyzes changed files, traces dependency graph, identifies all projects that depend on changed files.

13. **How do you migrate from another monorepo tool to Nx?**

    - Answer: Use migration guides, incrementally adopt Nx features, keep existing structure initially.

14. **What is the impact of Nx on CI/CD?**

    - Answer: Faster CI with affected commands, remote caching, parallel execution, but requires setup.

15. **How do you handle Nx in large teams?**

    - Answer: Clear ownership with tags, documentation, generators for consistency, Nx Cloud for cache sharing.

16. **Explain Nx's plugin system**

    - Answer: Plugins provide generators and executors for specific frameworks, extending Nx's capabilities.

17. **How do you debug Nx cache issues?**

    - Answer: Use `--skip-cache`, check `nx.json` configuration, verify inputs/outputs, use Nx Cloud analytics.

18. **What is the difference between executors and runners?**

    - Answer: Executors run tasks for projects; runners execute task graphs (e.g., Nx Cloud runner).

19. **How do you handle custom build processes in Nx?**

    - Answer: Create custom executors or use existing ones with configuration.

20. **How do you optimize Nx for very large monorepos?**

    - Answer: Use affected commands, remote caching, distributed execution, and proper project structure.

### FAANG-style (5)
21. **Design an Nx workspace for a microservices architecture**

    - Answer: Each service as a project, shared libs for common code, tags for boundaries, Nx Cloud for distributed builds.

22. **How would you implement Nx at scale with 100+ projects?**

    - Answer: Proper project structure, affected commands, remote caching, distributed execution, monitoring.

23. **Explain Nx's architecture and extensibility**

    - Answer: Core graph analysis, plugin system for generators/executors, task runner for execution, cache for optimization.

24. **How do you handle Nx with multiple programming languages?**

    - Answer: Use language-specific plugins, configure appropriate executors, maintain consistent structure.

25. **Design a CI/CD pipeline with Nx**

    - Answer: Detect changes with affected, run affected tasks, build with cache, deploy changed projects.

### Follow-ups (5)
26. **How do you handle Nx with Docker?**

    - Answer: Use `nx affected` to build only changed images, multi-stage Dockerfiles, cache Docker layers.

27. **What is the impact of Nx on developer experience?**

    - Answer: Faster builds, better tooling, consistent patterns, but learning curve.

28. **How do you handle Nx in a polyglot monorepo?**

    - Answer: Use appropriate plugins, configure executors per language, maintain consistent project structure.

29. **How do you handle Nx with different frameworks?**

    - Answer: Use framework-specific plugins (Nx React, Nx Next.js), configure appropriate targets.

30. **What are the alternatives to Nx?**

    - Answer: Turborepo, Lerna, Rush, or custom tooling with npm/yarn/pnpm workspaces.

## Summary

Nx provides a comprehensive monorepo solution with generators, affected analysis, computation caching, and dependency graph visualization. Master its features to build scalable, maintainable monorepos.
